# Yaksha-Prashna

📄 Paper: [link](https://drive.google.com/file/d/18AOocdUzzvs8qunbnF7SQbh1VwYcCd5D/view?usp=sharing)

## Overview

Many cloud infrastructure organizations increasingly rely on third-party eBPF-based network functions (NFs) for use cases such as security, observability, and load balancing. However, developing eBPF solutions requires highly skilled developers, which forces many organizations to depend on NFs from third-party marketplaces (e.g., L3AF). These NFs are typically distributed as bytecode, providing little visibility into their internal implementation. As a result, the bytecode may introduce security risks such as malware, spoofing (e.g., modifying the packet source IP to remain anonymous), or data exfiltration (e.g., cloning packets). In addition, the NFs may be functionally incorrect, which operators must detect before deployment. Ensuring both the security and functional correctness aspects of such bytecode is challenging due to the lack of transparency into its internal behavior.

In this work, we aim to enable operators to validate the security and functionality of third-party network function bytecode. We present a system, **Yaksha-Prashna**, that analyzes bytecode using a classic dataflow analysis framework to extract network context (NC), such as accessed protocols, header field modifications, and map read/write operations. Yaksha-Prashna also introduces a language that allows operators to express verification properties as queries. These queries are evaluated against the extracted network details to verify the correctness of the network function.

This repository contains the prototype implementation of Yaksha-Prashna.

The workflow: <br>
(1) For a given bytecode (optionally accompanied by NF chain information, if the user wants to check for unintended interactions before deploying a new NF), the analyzer builds a formal dataflow analysis model guided by control flow analysis. This model is represented as a control flow graph (CFG) and annotated with the network context (NC) extracted from the bytecode, resulting in (2) the CFG-NC. We define the bytecode’s network context as the set of read/write/copy operations on packet content (headers and metadata), the network protocols processed, the eBPF maps accessed, and the helper functions invoked by the bytecode. (3) The Query Engine (QE) translates the network context encoded in the CFG-NC into (4) the Knowledge Base (Prolog facts), while preserving the structural integrity of the CFG-NC. (5A) Assertions written in the proposed language are evaluated by the QE by (6) mapping their predicates to the corresponding facts stored in the Knowledge Base. (7A) If an assertion violation is detected, it is reported; otherwise, the NF can be deployed at the eBPF hook point. In the case of a violation, (5Q) retrieval queries are issued to the QE, and (8Q) the behavior of the NF or NF chain is retrieved.

---

### The Analyzer

The primary objective of the Analyzer is to extract network context from eBPF bytecode. However, performing this extraction at the bytecode level introduces two key challenges: (1) tracking data dependencies between registers and memory, and (2) leveraging this information to map low-level bytecode instructions to high-level network semantics. To address the first challenge, the Analyzer transforms the raw bytecode into a Control Flow Graph (CFG) and applies classical dataflow analysis to track data dependencies across the program. The resulting analysis maintains Register (R) and Memory (M) states that capture the evolving program state. These R and M states are subsequently used by a set of 15 transfer-function rules (Table 1 in the paper) to map bytecode instructions to their corresponding network context, thereby addressing the second challenge. Finally, the extracted NC is annotated onto the CFG at the basic-block level, producing an NC-augmented control-flow graph (CFG-NC).

---

### Query Engine

The Query Engine (QE) has two primary responsibilities:

1. Convert the CFG-NC generated by the Analyzer into a representation that can be reasoned about by the engine.
2. Verify the operator’s high-level intent against this representation.

---

#### Converting CFG-NC to Prolog Facts

The network context encoded in the CFG-NC is translated into **Prolog facts** while preserving the control-flow information of the CFG. This translation is performed by a module within the QE and follows a **deterministic one-to-one mapping** from CFG-NC constructs to Prolog facts. Further details of this mapping are described in **Section 5.1 of the paper**.

---

#### Verifying the Operator’s High-Level Intent

To enable verification, the operator’s high-level intent must be expressed in a form that can be queried over the generated Prolog facts. The system introduces the **Yaksha-Prashna language**, whose syntax is similar to that of Prolog. The language specification is shown in **Figure 2**.

Using this language, operators express verification properties that are evaluated by the QE.

For example, a common property of a firewall is:

> **A firewall should not modify packet contents.**

This property can be expressed using the following Yaksha-Prashna query:

> `!updatesField(xdp_fw, *).`

The query checks whether the network function updates any packet field. If such updates are detected, the query evaluates to **false**, indicating that the network function deviates from the expected behavior of a firewall.

## Folder Structure

The repository is organized as follows:

- `demo/`  
  Contains demonstrations showcasing Yaksha-Prashna's ability to detect and defend against malicious bytecode:
  - **Attack demonstration:** Examples of malicious network function bytecode that manipulates packet fields to evade security policies 
  - **Verifying NF:** Yaksha-Prashna detects these malicious behaviors

- `code/`  
  Contains the full implementation of Yaksha-Prashna.

Each directory includes its own README with instructions for building and running the corresponding components.
